## CRITICAL LABEL FORMAT RULE

**LOT MODE:** (01) GTIN + (10) Lot + (11) Manufacturing. NO expiration. Batch printing = N copies of the same label.
**SERIAL MODE:** (01) GTIN + (17) Expiration + (11) Manufacturing + (21) Serial. WITH expiration. Batch printing = sequential serial numbers.

**HRI DISPLAY ORDER (text printed on label):**
- LOT: `(01) GTIN` → `(10) LOT` → `(11) MfgDate`
- SERIAL: `(01) GTIN` → `(21) SERIAL` → `(11) MfgDate` → `(17) ExpDate`
- The identifier (lot or serial) is always the SECOND line, right after GTIN.
- This is different from the DataMatrix encoding order (which puts fixed-length AIs first).

## PHASE 7: BATCH / SEQUENTIAL PRINTING

### Step 7.1 — Print Job API

`server/src/routes/printJobs.routes.ts`:

Uses `printJobStore` (JsonStore reading from `data/print-jobs.json`):

- `POST /api/print-jobs` — Create and process print job:
  1. Validate inputs (Zod)
  2. Load template from `templateStore.findById()`
  3. **LOT MODE:** generate ZPL with `generateLotBatchZpl()`, total = copiesPerLabel. No serials.
  4. **SERIAL MODE with batch:** generate serials, then `generateBatchZpl()`. Includes expiration date.

- `GET /api/print-jobs` — paginated list from printJobStore. Support query filters: status, userId, dateFrom, dateTo.

### Step 8.4 — LAN Deployment

All users are on the same office WiFi/network. The app runs on the PC that has the Zebra GK420D connected via USB. Other users access it through that PC's local IP address. No cloud, no tunnels, no remote print queues.

**Server binding — `server/src/app.ts`:**

Bind to `0.0.0.0` (all network interfaces) so LAN devices can reach it:
```typescript
import os from 'os';

function getLocalIpAddress(): string {
  const interfaces = os.networkInterfaces();
  for (const iface of Object.values(interfaces)) {
    if (!iface) continue;
    for (const alias of iface) {
      if (alias.family === 'IPv4' && !alias.internal) return alias.address;
    }
  }
  return 'localhost';
}

const PORT = process.env.PORT || 3001;
const localIp = getLocalIpAddress();

app.listen(Number(PORT), '0.0.0.0', () => {
  console.log('');
  console.log('===========================================');
  console.log('  GS1 DataMatrix Label System — Running');
  console.log('===========================================');
  console.log(`  Local:   http://localhost:${PORT}`);
  console.log(`  Network: http://${localIp}:${PORT}`);
  console.log('');
  console.log('  Share the Network URL with other users');
  console.log('  on your office WiFi to give them access.');
  console.log('===========================================');
  console.log('');
});
```

**Printing architecture for LAN:**

Since all users are on the same network and Zebra Browser Print runs on the server PC:
- The server PC user prints directly — ZPL goes browser → Zebra Browser Print → USB printer.
- **Other LAN users** also need Zebra Browser Print installed on the server PC, but their browser communicates with it via `localhost`. HOWEVER — Zebra Browser Print only listens on `localhost` of the machine it's installed on. This means:
  - **LAN users CANNOT send ZPL directly to the server PC's Zebra Browser Print from their own browser.** The `fetch('http://localhost:9100/...')` call in their browser hits THEIR own localhost, not the server PC.
  - **Solution: Use the backend as a print proxy.** When a LAN user prints:
    1. Their browser sends the ZPL to the backend: `POST /api/print-jobs/:id/execute`
    2. The backend saves the ZPL and marks the job as `queued`
    3. On the server PC (where the printer is), a lightweight print polling loop checks for queued jobs every 5 seconds
    4. When a queued job is found, the server PC's browser (or a background script) sends the ZPL to Zebra Browser Print on localhost
    5. Job status updates to `completed`

Implement this print proxy in `server/src/services/printQueue.service.ts`:
```typescript
/**
 * Print queue for LAN users.
 *
 * Architecture:
 * 1. Any user (local or LAN) creates a print job via POST /api/print-jobs
 * 2. The ZPL is stored in the print job record (in data/print-jobs.json)
 * 3. On the Label Creator page, add a "Print Queue" panel visible on ALL clients
 * 4. The server PC user (who has the USB printer) sees queued jobs and clicks
 *    "Print Now" which sends the ZPL to Zebra Browser Print from THEIR browser
 * 5. Status updates from 'queued' → 'printing' → 'completed'
 *
 * This avoids the localhost restriction of Zebra Browser Print — only the
 * browser on the server PC talks to Zebra Browser Print.
 */
```

**Print Queue UI component** (`client/src/components/PrintQueue/PrintQueue.tsx`):
- Shown on the main page (label creator / dashboard)
- Polls `GET /api/print-jobs?status=queued` every 5 seconds
- Displays queued jobs: job ID, user who queued it, template, GTIN, label count, timestamp
- "Print Now" button per job — only functional if Zebra Browser Print is detected on THIS browser's machine (i.e., the server PC)
- "Print All Queued" button to process all queued jobs sequentially
- On non-printer PCs, the "Print Now" button is disabled with tooltip: "Connect to the printer PC to print, or ask the printer operator to process the queue"
- After clicking "Print Now": sends ZPL to Zebra Browser Print, updates job status via PATCH
