# AGENT 5 — QA: Backend, Data Integrity, GS1 Encoding & Print Pipeline

You are a QA engineer focused on backend correctness, data integrity, GS1 compliance, and the print pipeline. Your primary concern is that the system produces ACCURATE, SCANNABLE, REGULATORY-COMPLIANT labels.

Your job: test every API endpoint, verify GS1 encoding correctness, validate ZPL output, check data integrity in JSON files, and stress-test the system. Produce a structured bug report.

Read the full codebase before starting. Run the application. You will use a combination of: direct API calls (curl or Postman/Insomnia), reading JSON data files, inspecting generated ZPL, and testing barcodes on labelary.com.

---

## THE #1 RULE TO VERIFY

**LOT MODE: DataMatrix encodes (01)(11)(10) — NO AI 17 expiration. ZPL generates 3 HRI lines.**
**SERIAL MODE: DataMatrix encodes (01)(17)(11)(21) — WITH AI 17 expiration. ZPL generates 4 HRI lines.**

Every test involving GS1 encoding or ZPL output MUST verify this rule.

### T1.4 — HRI Text Generation (LINE ORDER IS CRITICAL)

**LOT MODE HRI:**
- [ ] Returns exactly 3 strings
- [ ] Line 1 = "(01) 00850008393006" (GTIN)
- [ ] Line 2 = "(10) MG96CI-260116" (Lot — the IDENTIFIER, not a date)
- [ ] Line 3 = "(11) 260116" (Manufacturing date)
- [ ] No string contains "(17)"
- [ ] **Order check:** line 2 is ALWAYS the identifier (10), line 3 is ALWAYS the date (11)

**SERIAL MODE HRI:**
- [ ] Returns exactly 4 strings
- [ ] Line 1 = "(01) 00850008393006" (GTIN)
- [ ] Line 2 = "(21) BWIII2026-1234" (Serial — the IDENTIFIER, not a date)
- [ ] Line 3 = "(11) 260218" (Manufacturing date)
- [ ] Line 4 = "(17) 330218" (Expiration date)
- [ ] **Order check:** line 2 is ALWAYS the identifier (21), lines 3-4 are ALWAYS dates

**Encoding order vs display order (CRITICAL distinction):**
- [ ] `buildGS1ElementString()` for LOT → produces `(01)...(11)...(10)...` (fixed AIs first, variable last = ENCODING order)
- [ ] `buildHriText()` for LOT → produces `(01)..., (10)..., (11)...` (identifier after GTIN = DISPLAY order)
- [ ] These two orders are INTENTIONALLY DIFFERENT — if they are the same, verify this is correct
- [ ] `buildGS1ElementString()` for SERIAL → produces `(01)...(17)...(11)...(21)...` (encoding order)
- [ ] `buildHriText()` for SERIAL → produces `(01)..., (21)..., (11)..., (17)...` (display order)
- [ ] HRI dates use YYMMDD format

**Serial number format validation:**
- [ ] Serial "BWIII2026-1234" → accepted (letters, digits, hyphens)
- [ ] Serial "ABC0001" → accepted
- [ ] Serial "SN-999" → accepted
- [ ] Serial with 20 characters → accepted (max length)
- [ ] Serial with 21 characters → rejected (not YYYY-MM-DD)

### T1.5 — Serial Number Generator
- [ ] prefix="ABC", start=234, end=298, step=1, pad=0 → 65 items, first="ABC234", last="ABC298"
- [ ] prefix="", suffix="-X", start=1, end=50, step=1, pad=4 → 50 items, first="0001-X", last="0050-X"
- [ ] step=5, start=1, end=50 → items=[1,6,11,16,21,26,31,36,41,46] → 10 items
- [ ] start=1, end=1, step=1 → 1 item
- [ ] start > end → should return empty or throw error (document which)
- [ ] step=0 → should throw or return error

### T3.3 — ZPL Coordinate Accuracy & Unit Conversion
- [ ] Element at (2mm, 2mm) at 203 DPI → ^FO16,16 (2 × 203/25.4 = 15.98 ≈ 16)
- [ ] Label 50.8mm wide → ^PW406 (50.8 × 203/25.4 = 406)
- [ ] Label 25.4mm tall → ^LL203 (25.4 × 203/25.4 = 203)
- [ ] Verify math: mmToDots(mm, 203) = Math.round(mm * 203 / 25.4)
- [ ] **Square label** 25.4mm × 25.4mm → ^PW203 and ^LL203 (equal width/height)
- [ ] **Large label** 101.6mm × 76.2mm → ^PW812 and ^LL609
- [ ] **Unit conversion round-trip:** 2 inches → 50.8mm → 406 dots at 203 DPI → verify all three conversions are correct
- [ ] **Inches input stored as mm:** If template was created with 2" × 1", verify templates.json has widthMm=50.8, heightMm=25.4 (NOT 2 and 1)
- [ ] **Margin field:** template with marginMm=1 → this value is stored but does NOT affect ZPL output (margin is a design-time visual guide, not a print command — the ZPL uses element positions as-is)
